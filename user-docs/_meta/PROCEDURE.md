# Процедура створення документації (Workflow)

Цей документ є фінальним і повним стандартом створення та підтримки технічної документації проєкту. Будь-які зміни в процесі мають лише доповнювати цей документ, не видаляючи існуючі структурні вимоги.

## 1. Дослідження (Research)
- **Аналіз коду:** Знайти основний React-компонент сторінки, його хуки, сервіси, константи та React Query запити.
- **Вкладки (Tabs) як Міні-сторінки:** Якщо сторінка містить таби, кожна вкладка аналізується як окрема функціональна одиниця зі своєю логікою та кодом.
- **Архітектурний стек:** Визначити, як дані подорожують від API через сервіси та хуки до інтерфейсу.
- **Прихована логіка:** Звірити код із `PAGES_DESCRIPTION.md` та `data/` для виявлення бізнес-правил, які не очевидні з UI.

## 2. Візуальна стратегія та Пріоритезація (ОБОВ'ЯЗКОВО)
Перед написанням тексту скласти повний план скріншотів та **негайно** зареєструвати їх у `_meta/SCREENSHOTS_INDEX.md` зі статусом `📝 Planned`.
**Обов'язкові типи та ракурси:**
1. **Full Page (`page_`):** 
   - Основний вигляд (main).
   - Окремий скріншот для **кожної вкладки** (tab).
   - Стан порожніх даних (empty).
   - Мобільна версія інтерфейсу (mobile).
2. **Modal (`modal_`):**
   - Чиста форма (form).
   - Детальна інформація (details).
   - Приклади помилок валідації (valid).
   - Вікна фільтрації (filters).
   - Попередження та конфлікти (force/conflict).
3. **Section (`sec_`):** Специфічні блоки (віджети статистики, графіки, складні елементи таблиць).

## 3. Створення контенту (Execution)

### Для Сторінки:
- Створити файл у `data/[розділ]/[ім'я].md`.
- Використовувати сухий технічний стиль (факти замість описів).
- **Вкладки:** Кожна вкладка повинна мати свій розгорнутий розділ другого рівня (`##`) з описом функціоналу та власним плейсхолдером скріншота.
- **ОБОВ'ЯЗКОВО** додати розділ "Технічна інформація" в кінці з повною картою файлів.

### Для Модального вікна:
- **Перевірка:** Перед створенням перевірити `data/shared/modals/`. Якщо файл є — актуалізувати його код та дату.
- **Реєстрація:** Оновити `_meta/MODALS.md` з актуальною датою.
- **Зміст:** Описати всі поля, їх типи, залежності та правила валідації.

## 4. Метадані та Синхронізація
- Оновити статус у `_meta/INDEX.md` на `✅ Completed`.
- **Дати:** Проставити актуальну дату оновлення (YYYY-MM-DD) у ВСІХ реєстрах (`INDEX.md`, `MODALS.md`, `SCREENSHOTS_INDEX.md`) для кожного зміненого запису.

## СТРУКТУРА ТЕХНІЧНОГО ОПИСУ СТОРІНКИ
1. **Frontmatter** (route, permission, components: список ключових файлів).
2. **# Заголовок сторінки**.
3. `{{PAGE:main_screenshot_id}}` (Головний скріншот).
4. **Призначення** (1-2 речення).
5. **Вкладки (Tabs) / Розділи**:
   - **## Назва вкладки**.
   - `{{PAGE:tab_screenshot_id}}`.
   - Детальний опис функціоналу, унікальних кнопок та логіки саме цієї вкладки.
6. **Функціональні можливості** (Загальний технічний список можливостей сторінки).
7. **Реєстр / Таблиця даних** (Таблиця: Колонка | Опис | Логіка/Сортування).
8. **Операційне управління** (Опис усіх кнопок дій: Глобальні та Рядкові).
9. **Технічна інформація (для розробників)** (Таблиця: Файл | Роль: компоненти, хуки, запити, сервіси, константи).
10. **Технічні особливості** (Логіка валідації, автоматична синхронізація, нюанси адаптивності, робота з localStorage).

## СТРУКТУРА ОПИСУ МОДАЛЬНОГО ВІКНА
1. **# Заголовок модалки**.
2. `{{MODAL:screenshot_id}}`.
3. **Призначення** (Для чого це вікно).
4. **Поля форми / Вміст** (Таблиця: Поле | Опис | Технічні деталі: типи вводу, обов'язковість, динамізм).
5. **Логіка та валідація** (Залежності між полями, правила збереження, обробка статусів 409).
6. **Технічна інформація** (Таблиця: Файл | Роль — компоненти, хуки, API методи).
