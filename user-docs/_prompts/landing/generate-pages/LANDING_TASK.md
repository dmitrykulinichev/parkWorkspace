## Алгоритм

1. Прочитати файл стилю (вказаний в промпті) — він визначає тон і подачу
2. Прочитати вихідний файл документації
3. Знайти всі `{{MODAL:doc_id}}` → прочитати відповідні файли з `user-docs/data/shared/modals/`
4. Знайти всі `{{SCRN:id}}` → обробити скріншоти (див. нижче)
5. Сформувати вихідний файл за структурою нижче
6. Записати у `landing/www/content/docs/[ім'я вказане в промпті]`
7. Якщо файл уже існує — прочитати його та доповнити, а не перезаписати
8. Якщо в промпті вказані вкладки — згенерувати файл лендінгу для кожної (див. нижче)
9. Оновити `C:\Repository\park\landing\www\content\docs\menu.json` (див. нижче)

## Вкладки лендінгу

**Якщо в промпті немає поля "Вкладки для лендінгу" або воно порожнє** — вкладки не генерувати. Якщо результуючий файл вже існує — весь його поточний вміст зберегти без змін.

**Якщо вкладки вказані** — для кожної:

1. Визначити файли:
   - **Джерело:** та сама папка що й батьківський файл джерела + той самий числовий префікс + суфікс з промпту

2. Прочитати файл-джерело вкладки, трансформувати за тими самими правилами що й батьківська сторінка (стиль, скріншоти, модалки)

3. Якщо файл результату вже існує — прочитати і доповнити, не перезаписувати

4. Записати результат

## Оновлення menu.json

Після запису файлу — оновити `landing/www/content/docs/menu.json`.

**Обов'язковий порядок:** `read_file` → модифікація в пам'яті → `write_file`. Не редагувати частинами.

**Алгоритм:**

1. Прочитати `menu.json`
2. Знайти запис де `slug` = ім'я файлу без `.md`
   - **Знайдено** → оновити `image` якщо перший скріншот сторінки змінився; `title` і `group` не чіпати
   - **Не знайдено** → додати новий запис у групу, вказану в промпті

**Формат запису без вкладок:**

```json
{ "title": "[H1 заголовок]", "slug": "[ім'я файлу без .md]", "image": "[перший скріншот або порожній рядок]" }
```

**Формат запису з вкладками:**

```json
{
  "title": "[H1 заголовок]",
  "slug": "[ім'я файлу без .md]",
  "image": "[перший скріншот або порожній рядок]",
  "tabs": [
    { "title": "[Назва вкладки]", "slug": "[reports--maintenance]" },
    { "title": "[Назва вкладки]", "slug": "[reports--performance]" }
  ]
}
```

Якщо запис батьківської сторінки вже існує і вкладки згенеровані — додати або оновити масив `tabs` у цьому записі. Наявні вкладки в масиві не видаляти.

Якщо в промпті група не вказана і slug не знайдено — залишити `menu.json` без змін і повідомити що запис потрібно додати вручну.

**Завжди прибирати:** frontmatter (`route`, `permission`, `components`), розділи "Технічна інформація" та "Технічні особливості", шляхи до файлів коду, назви хуків/сервісів, permission-рядки, будь-який код.

**Включати:** назву, призначення, функціональні можливості, вкладки, дії користувача, ключові поля форм — все переформульоване мовою переваг згідно з файлом стилю.

## Фаза скріншотів

Для кожного `{{SCRN:id}}` у вихідному файлі:

### 1. Визначити шлях до файлу

| Префікс id | Папка в user-docs |
| :--- | :--- |
| `page_` (без `_mobile`)  | `user-docs/screenshots/desktop/pages/`   |
| `page_*_mobile`          | `user-docs/screenshots/mobile/pages/`    |
| `tab_` (без `_mobile`)   | `user-docs/screenshots/desktop/tabs/`    |
| `tab_*_mobile`           | `user-docs/screenshots/mobile/tabs/`     |
| `modal_` (без `_mobile`) | `user-docs/screenshots/desktop/modals/`  |
| `modal_*_mobile`         | `user-docs/screenshots/mobile/modals/`   |
| `sec_` (без `_mobile`)   | `user-docs/screenshots/desktop/sections/`|
| `sec_*_mobile`           | `user-docs/screenshots/mobile/sections/` |

### 2. Перевірити наявність і скопіювати

**Файл існує** → скопіювати до `landing/www/assets/img/docs/[id].png`
- Копіювати лише світлу тему (без `_dark`)
- Завжди перезаписувати — файл міг оновитись

**Файл не існує** → нічого не копіювати. Shortcode однаково вставляти (сайт покаже placeholder автоматично).

### 3. Результат у тексті

Завжди вставляти shortcode — незалежно від наявності файлу:
```
{{screenshot file="[id].png" title="[описовий заголовок українською]"}}
```

Якщо файл відсутній — сайт відобразить заглушку «Скріншот не додано» до появи реального зображення.

**Скільки скріншотів вставляти:** обирай 1–3 найбільш показових. Кілька сильних ілюстрацій кращі за десяток формальних.

## Трансформація модалок

`{{MODAL:doc_id}}` в тексті — не переносити як плейсхолдер. Замість цього:
- Прочитати файл модалки з `user-docs/data/shared/modals/[doc_id].md`
- Описати як крок у процесі або підрозділ — мовою користувача, без технічних деталей

## Структура вихідного файлу

Структура **не копіює** технічну документацію — будується навколо переваг. Маєш свободу в порядку секцій. Єдине правило: від загального до конкретного.

Орієнтовний шаблон:

```markdown
# [Назва — коротко і ємко]

[Вступ: 1–2 речення про цінність для бізнесу, не про функції]

{{screenshot file="page_[...]_main.png" title="[Назва]"}}

## [Головна перевага або ключовий сценарій]

[2–4 речення або маркований список. Кожен пункт — конкретна перевага.]

## [Інша перевага або вкладка]

{{screenshot file="[...]" title="[...]"}}

[...]

## Як це працює

[Ключові дії користувача. Модалки — як кроки в процесі, не як технічні форми.]
```
