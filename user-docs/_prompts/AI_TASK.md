# Процедура створення документації (AI Workflow)

> Цей файл описує **універсальний процес і структуру** документації.
> Перед початком роботи прочитати всі три файли: `_prompts/PROJECT.md` (налаштування проєкту), `_prompts/STYLE_GUIDE.md` (конвенції іменування), цей файл (workflow).

---

## Загальні вимоги

- **Мова:** Визначена в `PROJECT.md`. Всі тексти, заголовки, описи — відповідно до неї.
- **Тон:** Сухий технічний стиль. Факти, не описи. "Сторінка дозволяє керувати..." замість "Зручна сторінка для управління...".
- **Повнота:** Описати всі поля форм, всі кнопки дій, всі стани інтерфейсу (empty, error, mobile).
- **Заборонено:** пропускати розділи, змінювати їхні назви або порядок, додавати розділи поза шаблоном.

### Редагування існуючих файлів

При будь-якому оновленні файлів — реєстрів `_meta/`, модалок, сторінок — **завжди читати файл повністю перед змінами**.

**Обов'язковий порядок:** `read_file` → модифікація в пам'яті → `write_file` (повний перезапис).

Часткове редагування через `replace` заборонено: призводить до мовчазної втрати наявного вмісту.

---

### Робота з наявними файлами документації

Перед створенням будь-якого файлу — перевірити чи він вже існує.

**Якщо файл існує:**
1. Прочитати повністю.
2. Порівняти з кодом: знайти застарілі дані, відсутні поля, неточності.
3. **Доповнити та скорегувати** наявний вміст — зберігаючи все коректне.
4. Не переписувати розділи, які вже відповідають дійсності.

**Якщо файл не існує** — створити з нуля за шаблоном нижче.

> Мета — акумулювати знання, а не замінювати їх. Кожна ітерація має робити документ повнішим, а не іншим.

## Фаза 1: Дослідження коду

Перед написанням тексту — дослідити код цільової сторінки.
Структура коду та стек — у `PROJECT.md`.

**Якщо задача вказана через URL** (наприклад `/drivers`):
1. Знайти маршрут у файлі роутів (див. `PROJECT.md`) → отримати назву компонента
2. Знайти файл компонента за назвою у папці сторінок
3. Далі — стандартний аналіз нижче

**Якщо задача вказана через шлях до файлу** — починати відразу з пункту 1 нижче.

1. Знайти головний компонент/модуль сторінки
2. Визначити всі модулі логіки та стану (хуки, store, composables — згідно зі стеком) та їх відповідальності
3. Знайти API-клієнт/сервіс і перелічити всі методи (`create`, `update`, `delete`, `get`, `export` тощо)
4. Знайти запити серверного стану/кешу та їх ключі
5. Знайти константи та enum-и, що визначають бізнес-логіку (типи, статуси)
6. **Якщо в промпті вказані вкладки ("Вкладки на сторінці"):** аналізувати кожну вкладку як окремий компонент. Якщо поле відсутнє або порожнє — вкладки не аналізувати
7. Виявити бізнес-правила, не очевидні з UI: автоматичні тригери, залежності між модулями, побічні ефекти

## Фаза 2: Планування скріншотів (ОБОВ'ЯЗКОВО ДО НАПИСАННЯ)

1. Скласти повний список скріншотів для цієї сторінки/модалки
2. Іменувати за правилом: `[тип]_[модуль]_[об'єкт]_[стан]` (деталі — `_prompts/STYLE_GUIDE.md`)
3. **Зареєструвати кожен** в `_meta/SCREENSHOTS.md` зі статусом `📝 Planned` і поточною датою
4. **Одночасно** — заповнити інструкції захоплення в `_meta/RECORDER.md` (див. нижче)

**Мінімальний набір для сторінки:**

| ID | Коли обов'язковий |
| :--- | :--- |
| `page_[module]_[name]_main` | Завжди |
| `page_[module]_[name]_main_dark` | Якщо продукт має темну тему (опціонально) |
| `page_[module]_[name]_empty` | Якщо є empty-стан |
| `page_[module]_[name]_mobile` | Якщо є мобільна версія (зберігати в `screenshots/mobile/`) |
| `tab_[module]_[page]_[tabname]` | Для **кожної** вкладки окремо |
| `tab_[module]_[page]_[tabname]_dark` | Темна тема вкладки (опціонально) |

Плейсхолдер у тексті для всіх скріншотів: `{{SCRN:[id]}}`

**Мінімальний набір для модального вікна:**

| ID | Коли обов'язковий |
| :--- | :--- |
| `modal_[module]_[name]_form` | Якщо є форма введення |
| `modal_[module]_[name]_form_dark` | Темна тема форми (опціонально) |
| `modal_[module]_[name]_valid` | Якщо є поля з валідацією |
| `modal_[module]_[name]_details` | Якщо є режим перегляду |
| `modal_[module]_[name]_filters` | Якщо є фільтри |

Плейсхолдер у тексті для всіх скріншотів: `{{SCRN:[id]}}`

### Генерація Katalon-скриптів

Для **кожного** запланованого скріншота — створити файл `user-docs/katalon/[id].side`.

Робити під час аналізу коду: CSS-селектори видно у JSX, `onClick` обробниках, умовних рендерах — пізніше цього контексту не буде.

**Що витягати з коду:**

| Що потрібно | Де шукати в коді |
| :--- | :--- |
| Головний контейнер сторінки | кореневий `<div>` компонента або `data-testid` |
| Кнопка відкриття модалки | `onClick` → `setOpen(true)` або аналог |
| Контейнер модалки | `[role="dialog"]` або з `PROJECT.md` |
| Кнопка вкладки | `<Tab>` або `[role="tab"]` |
| Контент вкладки | `[role="tabpanel"]` або з `PROJECT.md` |

Якщо точний селектор невідомий — використовувати семантичні з `PROJECT.md` (розділ "Katalon — CSS-селектори рівня проєкту").

**Шаблони команд по типах:**

`page_*_main` / `page_*_empty`:
```
open                    → /[url]
waitForElementPresent   → css=[main-container]   | 10000
captureEntirePageScreenshot → [output-path]
```

`page_*_mobile`:
```
setWindowSize           → 375x812
open                    → /[url]
waitForElementPresent   → css=[main-container]   | 10000
captureEntirePageScreenshot → [output-path]
setWindowSize           → 1920x1080
```

`tab_*`:
```
open                    → /[url]
waitForElementPresent   → css=[role="tablist"]   | 10000
click                   → css=[tab-button]
waitForElementPresent   → css=[role="tabpanel"]:not([hidden]) | 5000
captureEntirePageScreenshot → [output-path]
```

`modal_*`:
```
open                    → /[url]
waitForElementPresent   → css=[page-content]     | 10000
click                   → css=[trigger-button]
waitForElementPresent   → css=[role="dialog"]    | 5000
captureEntirePageScreenshot → [output-path]
```

**Шляхи збереження (output-path):**

| Тип | Шлях |
| :--- | :--- |
| `page_*` (не mobile) | `screenshots/pages/[id].png` |
| `page_*_mobile` | `screenshots/mobile/[id].png` |
| `tab_*` | `screenshots/tabs/[id].png` |
| `modal_*` | `screenshots/modals/[id].png` |
| `sec_*` | `screenshots/sections/[id].png` |

**Повний формат `.side` файлу:**

```json
{
  "id": "[id]",
  "version": "2.0",
  "name": "[id]",
  "url": "[base_url з PROJECT.md]",
  "tests": [{
    "id": "[id]-test",
    "name": "capture",
    "commands": [
      { "id": "1", "command": "open", "target": "/[url]", "value": "" },
      { "id": "2", "command": "waitForElementPresent", "target": "css=[selector]", "value": "10000" },
      { "id": "3", "command": "captureEntirePageScreenshot", "target": "[output-path]", "value": "" }
    ]
  }],
  "suites": [{
    "id": "[id]-suite",
    "name": "default",
    "persistSession": false,
    "parallel": false,
    "timeout": 300,
    "tests": ["[id]-test"]
  }],
  "urls": [],
  "plugins": []
}
```

**Після створення `.side` файлу** — додати рядок в індекс `_meta/RECORDER.md`:

```
| `[id]` | `katalon/[id].side` | [нотатка або —] |
```

Нотатка потрібна лише якщо є умова що не автоматизується (наприклад: "⚠️ потребує порожньої БД").

## Фаза 3: Структура файлу СТОРІНКИ

**Файл:** `workspace/user-docs/data/[розділ]/[назва].md`

**Правило:** дотримуватись точної послідовності та точних назв `##`-розділів. Порядок незмінний.

---

```
---
[frontmatter поля згідно з PROJECT.md]
---

# [Назва сторінки]

{{SCRN:[page_module_name_main]}}

> [Призначення — одне речення: що ця сторінка дає користувачу і навіщо вона існує.]
```

---

### БЛОК B — Огляд (завжди)

```
## Огляд

[2–4 речення. Яку частину бізнес-процесу покриває. З якими даними працює.
Чи залежить від інших модулів системи.]
```

---

### БЛОК C — Вкладки (тільки якщо на сторінці є tabs)

**Якщо в промпті немає поля "Вкладки на сторінці" або воно порожнє** — Блок C пропустити повністю. Не аналізувати вкладки, не створювати файли вкладок.

**Якщо результуючий файл вже існує і містить розділ `## Вкладки`** — зберегти його вміст без змін. Доповнювати лише якщо в промпті вказані нові вкладки яких там ще немає.

Кожна вкладка — **окремий файл** поруч з батьківською сторінкою (структура — Фаза 3б).

Файл батьківської сторінки містить лише перелік вкладок з посиланнями і одним реченням про кожну.

**Іменування файлу вкладки:** ім'я батьківського файлу (без `.md`) + суфікс з промпту.

Приклад: батько `05-reports.md` + суфікс `--maint.md` → файл `05-reports--maint.md` в тій самій папці.

```
## Вкладки

| Вкладка | Файл | Призначення |
| :--- | :--- | :--- |
| [Назва Вкладки 1] | [05-reports--maint.md](05-reports--maint.md) | [Одне речення] |
| [Назва Вкладки 2] | [05-reports--performance.md](05-reports--performance.md) | [Одне речення] |
```

Блоки E, F, G, H у файлі батьківської сторінки описують **тільки те, що спільне для всієї сторінки** незалежно від активної вкладки.

## Фаза 3б: Структура файлу ВКЛАДКИ

**Файл:** `data/[розділ]/[N]-[page]--[tab-slug].md`

**Правило:** структура ідентична файлу сторінки (Блоки A–H) з трьома відмінностями:
1. Frontmatter має додаткові поля `tab` і `parent`
2. Блок A використовує `{{SCRN:tab_...}}` замість `{{SCRN:page_...}}`
3. Блок C (Вкладки) не використовується — вкладки в вкладках не допускаються

### БЛОК A — Шапка вкладки (завжди)

```
---
route: "/[url-батьківської-сторінки]"
tab: "[Назва вкладки як відображається в UI]"
parent: "[N]-[page].md"
permission: "..."
components:
  - "..."
---

# [Назва вкладки]

{{SCRN:tab_[module]_[page]_[tabname]}}

> [Призначення — одне речення: що показує ця вкладка і навіщо вона.]
```

Блоки B, D, E, F, G, H — за шаблоном сторінки. Заповнювати лише ті, що актуальні для цієї вкладки; порожні пропускати.

**Після запису файлу вкладки** — зареєструвати в `_meta/INDEX.md` з типом `tab` і посиланням на батька (деталі — Фаза 5).

## Фаза 4: Документування модальних вікон

> **Перед записом кожного файлу:** `read_file` → модифікація в пам'яті → `write_file`. Replace заборонений.

**Ключовий принцип:** кожне модальне вікно, знайдене на сторінці, — це **окремий файл** у `data/shared/modals/`. Модалки є спільними між сторінками, тому зберігаються централізовано, а не всередині файлу сторінки.

**Алгоритм для кожної модалки на сторінці:**

1. Знайти модалку в коді сторінки
2. Перевірити чи існує `data/shared/modals/[modal_doc_id].md`
   - **Існує** → прочитати повністю, звірити з кодом, доповнити відсутні поля/логіку, скорегувати неточності. Наявний коректний вміст зберігати.
   - **Не існує** → створити з нуля за шаблоном нижче
3. Перевірити `_meta/MODALS.md` — чи є цей modal_doc_id у реєстрі
   - **Є** → у колонці "Використовується на сторінках" дописати поточну сторінку (якщо її ще немає), оновити дату
   - **Немає** → додати новий рядок з усіма колонками, включно з поточною сторінкою
4. Переконатись що в тексті файлу сторінки стоїть `{{MODAL:[modal_doc_id]}}` у потрібному місці

**Файл:** `data/shared/modals/[modal_doc_id].md`

**Правило:** дотримуватись точної послідовності та точних назв `##`-розділів.

---

```
{{SCRN:[modal_module_name_form]}}

> [Призначення — одне речення: для чого це вікно і звідки відкривається.]
```

## Фаза 5: Оновлення реєстрів

Виконати **після завершення** кожного документа:

> **Для кожного реєстру нижче:** `read_file` → модифікація в пам'яті → `write_file`. Replace заборонений — він мовчки обрізає існуючі записи.

**`_meta/INDEX.md`**
- Для сторінки: Статус → `✅ Completed`, Дата → `YYYY-MM-DD`, Тип → `page`, Батько → `—`
- Для кожної вкладки: додати окремий рядок з Тип → `tab`, Батько → посилання на батьківський файл

  | Розділ / Сторінка | Route | Файл документації | Тип | Батько | Статус | Дата |
  | `└ [Назва вкладки]` | `/[url]` | `[файл](../data/...)` | `tab` | `[батько](../data/...)` | `✅ Completed` | `YYYY-MM-DD` |

**`_meta/MODALS.md`**
- Для кожної модалки сторінки: перевірити чи поточна сторінка вказана у колонці "Використовується на сторінках"
- Якщо не вказана — дописати через кому
- Якщо модалки немає в реєстрі — додати повний рядок:

  | ID | Назва (UI) | Файл опису | Використовується на сторінках | Дата |
  | `modal_doc_id` | `Назва` | `[файл](../data/shared/modals/...)` | `поточна сторінка` | `YYYY-MM-DD` |

**`_meta/SCREENSHOTS.md`**
- Залишити статус `📝 Planned` (скріншоти ще не зроблені)

**`_meta/RECORDER.md`**
- Має бути заповнений ще на Фазі 2 — тут лише перевірити що кожен новий скріншот має рядок в індексі

**`katalon/`**
- Перевірити що для кожного нового скріншота є файл `katalon/[id].side`

## Фаза 6: Самоперевірка (обов'язково після написання)

Перечитай згенеровану документацію і виконай кожен пункт активно — не "здається що є", а перевір фактично.

**Змістова точність (звір з кодом):**
- [ ] Перечитай секцію **Поля форми** — відкрий компонент модалки і переконайся що кожне поле присутнє і тип вказаний правильно
- [ ] Перечитай **Управління (Actions)** — звір з кодом, чи не пропущена жодна кнопка
- [ ] Перечитай **Технічну інформацію** — перевір що всі ключові файли (хуки, сервіс, компонент) вказані
- [ ] Якщо є таби — перевір що кожен таб має окремий файл `[page]--[tab].md` і зареєстрований в INDEX.md з типом `tab`

**Структурна відповідність:**
- [ ] Усі обов'язкові блоки (A–H) присутні та в правильному порядку
- [ ] Frontmatter заповнений: `route`, `permission`, `components`
- [ ] Якщо файл існував — наявний коректний вміст збережений, не перезаписаний

**Консистентність реєстрів:**
- [ ] Усі `{{MODAL:doc_id}}` є ключами в `_meta/MODALS.md`
- [ ] Усі `{{SCRN:id}}` зареєстровані в `_meta/SCREENSHOTS.md`
- [ ] Для кожного нового `{{SCRN:id}}` є файл `katalon/[id].side` і рядок в `_meta/RECORDER.md`
- [ ] Кожна модалка сторінки має файл у `data/shared/modals/`
- [ ] В `_meta/MODALS.md` для кожної модалки поточна сторінка вказана у колонці "Використовується на сторінках"
- [ ] `_meta/INDEX.md` оновлений зі статусом `✅ Completed` і датою
