# План впровадження мультимовності (i18n)
_Створено: 2026-04-17. Використовувати як початковий контекст при поверненні до задачі._

---

## Рішення

Мультимовність розбита на два незалежних шари — інтерфейс і контент у БД. Вони підключаються і наповнюються незалежно одне від одного.

| Шар | Інструмент | Що покриває |
|-----|------------|-------------|
| UI (фронтенд) | `react-i18next` | Кнопки, меню, заголовки, форми, повідомлення про помилки |
| UI (бекенд) | Laravel `lang/` | Validation messages, Telegram/email сповіщення |
| Контент у БД | `spatie/laravel-translatable` | Назви планів, описи пермішенів, будь-який адмін-контент |

**Ключовий принцип:** підключення інфраструктури не вимагає одночасного перекладу всього. Після налаштування старий контент продовжує працювати як є — переклади додаються поступово.

**Мови:** українська (основна), англійська, російська.

---

## Порядок дій

### 1. Бекенд — Laravel `lang/`

- [ ] Встановити структуру директорій: `lang/uk/`, `lang/en/`, `lang/ru/`
- [ ] Перенести існуючі validation messages у `lang/uk/validation.php`
- [ ] Створити порожні `lang/en/validation.php`, `lang/ru/validation.php`
- [ ] Middleware що читає локаль з заголовку запиту (`Accept-Language`) або з профілю юзера і викликає `app()->setLocale()`
- [ ] В `UserResource` повертати поточну локаль юзера (зберігати в `users.locale`)

### 2. База даних — `spatie/laravel-translatable`

- [ ] `composer require spatie/laravel-translatable`
- [ ] Визначити які моделі потребують перекладу (пріоритет: `SubscriptionPlan`, `Permission`, описи фіч)
- [ ] Для кожної моделі: додати трейт `HasTranslations`, вказати `$translatable`, написати міграцію що конвертує рядкові поля в JSON
  ```php
  // Було: name = "Стандарт"
  // Стало: name = {"uk": "Стандарт", "en": "", "ru": ""}
  ```
- [ ] Налаштувати fallback локаль на `uk` (якщо переклад відсутній — показувати українською)
- [ ] **Робити поступово** — кожна модель окремою задачею, не все одразу

### 3. Фронтенд — `react-i18next`

- [ ] `npm install react-i18next i18next`
- [ ] Налаштувати `i18n.js` — ініціалізація, fallback `uk`, завантаження з JSON-файлів
- [ ] Підключити `I18nextProvider` в `app.jsx`
- [ ] Створити структуру файлів перекладів:
  ```
  resources/js/locales/
    uk.json
    en.json
    ru.json
  ```
- [ ] Визначити структуру ключів (важливо зробити одразу):
  ```json
  {
    "nav": { "drivers": "Водії", "vehicles": "Автомобілі" },
    "common": { "save": "Зберегти", "cancel": "Скасувати" },
    "auth": { "login": "Увійти", "logout": "Вийти" }
  }
  ```
- [ ] Зберігати вибрану мову в `localStorage` і в профілі юзера (`users.locale`)
- [ ] Додати перемикач мови у профіль (вкладка "Загальні дані")
- [ ] **Наповнення:** починати з `NavigationItems.js` і загальних UI-компонентів, далі поступово по сторінках

## Важливі рішення прийняті заздалегідь

**Структура ключів перекладів** — визначити до початку наповнення, щоб уникнути хаосу. Рекомендована структура по неймспейсах: `nav`, `common`, `auth`, `drivers`, `vehicles`, `finance` і т.д.

**Fallback завжди `uk`** — якщо ключ не перекладений, показується українська. Дозволяє випускати незавершені переклади без ризику порожніх рядків в UI.

**Нові фічі — одразу з `t('key')`** — після підключення інфраструктури будь-який новий текст пишеться через хелпер, а не хардкодом. Старий текст переїжджає поступово.

**Локаль зберігається на сервері** (`users.locale`) — щоб при наступному логіні або на іншому пристрої мова збереглася.

---

## Що НЕ потрібно робити одразу

- Переводити всі існуючі компоненти — це поступова робота
- Конвертувати всі моделі в `translatable` — тільки ті де є потреба
- Перекладати Telegram-повідомлення — низький пріоритет, переважно технічний контент
