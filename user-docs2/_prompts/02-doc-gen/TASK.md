# 02-doc-gen — Генерація документації елемента

## Вхідні дані

Промпт приймає **один i-doc ID**, наприклад:
- `page:tasks`
- `tab:reports-operational`
- `modal:task-form`
- `sec:task-list` *(секція документується у файлі батьківської сторінки або таба)*

## Алгоритм

### Крок 1. Знайти елемент у nav.json

Читаємо `_meta/nav.json` і знаходимо вузол з `idoc == <target>`.
З вузла беремо:
- `path` — URL (для pages)
- `label` — назва у навігації
- `hint` — підказка (дороговказ, не вичерпний опис)
- `docFile` — куди записати результат
- `tabs[]` — таби (є у page-вузлів)
- `sections[]` — секції
- `modals[]` — модалки

Для **tab**: шукаємо у батьківській page-ноді масив `tabs`, знаходимо потрібний за `idoc`.
Для **modal**: шукаємо у `modals[]` відповідної page або tab-ноди.

### Крок 2. Знайти вихідний код

З коментаря у реєстрі або по логіці назви:
- page: `resources/js/pages/<Name>/index.jsx` або `resources/js/pages/<Name>.jsx`
- tab: `resources/js/pages/<PageName>/<TabName>Tab.jsx`
- modal: `resources/js/components/modals/<ModalName>Modal.jsx`

Читаємо весь вихідний код компонента. Звертаємо увагу на:
- Заголовки та тексти в UI
- Назви секцій, кнопок, полів форми
- Які дані відображаються у таблицях
- Які фільтри/дії доступні
- Які модалки відкриваються і з яких дій

### Крок 3. Прочитати наявну документацію

Якщо `docFile` вже існує — прочитати перед написанням.
Мета: оновити, а не перезаписати — зберегти ручні правки де є.

### Крок 4. Написати документацію

Формат залежить від типу елемента — деталі нижче.
Писати **українською**. Мова — ділова, зрозуміла не-технічному користувачу.
Hint використовувати як **дороговказ** (загальний напрямок), не копіювати дослівно.

## Формати документів

### page без табів → `data/pages/page_X.md`

```markdown
# [Назва сторінки]
> `page:X` · `[URL]`

[2-3 речення: що це за сторінка, для кого вона, що тут можна зробити]

## Що тут є

[Загальний огляд вмісту — секції, ключові дії, таблиці]

## Розділи

### [Назва секції]
> `sec:X`

[Опис секції: що відображається, які дії доступні]

## Модальні вікна

### [Назва модалки]
> `modal:X` · `data/modals/modal_X.md`

[Одне речення: навіщо ця модалка. Детальна документація — у відповідному файлі.]
```

---

### page з табами → `data/pages/page_X.md`

Для сторінок де є `tabs[]` — page-файл є **навігаційним оглядом**, не документацією контенту.
Контент кожного таба — у відповідному `data/tabs/tab_X.md`.

```markdown
# [Назва сторінки]
> `page:X` · `[URL]`

[1-2 речення: що це за розділ системи]

## Вкладки

| Вкладка | За замовчуванням | Опис |
|---|---|---|
| [`tab:X-first`](../../tabs/tab_X_first.md) | ✓ | [Короткий опис] |
| [`tab:X-second`](../../tabs/tab_X_second.md) | | [Короткий опис] |

> Детальна документація кожної вкладки — у відповідних файлах.
```

---

### tab → `data/tabs/tab_X.md`

```markdown
# [Назва вкладки]
> `tab:X` · [`page:parent`](../pages/page_parent.md) · `[URL сторінки]`[` · за замовчуванням` — якщо isDefault]

[2-3 речення: що відображається в цій вкладці]

## Розділи

### [Назва секції]
> `sec:X`

[Опис: що за дані, які дії, фільтри, таблиці]

## Модальні вікна

### [Назва модалки]
> `modal:X`

[Одне речення. Посилання на modal-файл якщо є.]
```

---

### modal → `data/modals/modal_X.md`

```markdown
# [Назва модалки]
> `modal:X` · відкривається з: [`page:parent`](../pages/page_parent.md)

[1-2 речення: навіщо ця форма/вікно]

## Поля / Вміст

| Поле | Тип | Опис |
|---|---|---|
| [Назва поля] | текст / вибір / дата / ... | [що вводить користувач] |

## Дії

- **[Кнопка]** — [що відбувається]
```

---

## Правила

- Писати з точки зору користувача, не розробника
- Не описувати технічні деталі реалізації (React props, API-ендпоінти)
- Якщо компонент завантажує дані — описати **що** відображається, не як запит відбувається
- Якщо є перевірка прав доступу — згадати коротко які ролі мають доступ
- Для модалок-форм: перелічити всі поля та кнопки
- Для секцій-таблиць: описати які колонки відображаються і які дії є в рядку
