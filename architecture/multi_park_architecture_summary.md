# Резюме архітектурного обговорення: Мульти-парк та авторизація

## 1. Поточна архітектура
Система побудована за принципом **Single Database, Shared Schema**.
*   **Ізоляція:** Реалізована через Global Scope (`App\Models\Scopes\ParkScope`).
*   **Контекст:** Визначається Middleware `SetCurrentPark`, який зчитує `park_id` з моделі авторизованого користувача (`User`) і передає його в сервіс `CurrentPark`.
*   **User:** Жорстко прив'язаний до парку через колонку `park_id`.

## 2. Задача
Забезпечити можливість одному користувачу (наприклад, менеджеру) мати доступ до кількох парків і перемикатися між ними без необхідності створення окремих акаунтів.

## 3. Обране рішення: "Динамічне перемикання контексту"
Замість складної логіки сесій, ми вирішили використовувати **зміну стану в БД**.

**Суть методу:**
`park_id` у таблиці `users` виступає не як жорстка прив'язка, а як **"поточний активний контекст"**.
Для перемикання парку ми просто оновлюємо поле `park_id` у користувача.

**Переваги:**
1.  **Простота:** Не потрібно змінювати Middleware або логіку Scopes.
2.  **Сесії:** Авторизація (Sanctum) не злітає, оскільки `user_id` не змінюється.
3.  **Універсальність:** Механізм підходить і для менеджерів, і для трансферу водіїв/авто.

## 4. Критичний нюанс: Проблема "Привида" (Ghost User)
При зміні `park_id` користувач "зникає" з області видимості попереднього парку через роботу `ParkScope`.

**Сценарій проблеми:**
1.  Менеджер створює транзакцію в Парку А.
2.  Менеджер перемикається в Парк Б (його `park_id` стає Б).
3.  Адмін Парку А переглядає історію транзакцій.
4.  **Помилка:** Система не може завантажити автора транзакції, бо `ParkScope` шукає юзера в Парку А, а він вже в Парку Б.

**Рішення:**
Використання `withoutGlobalScope(ParkScope::class)` у зв'язках моделей, які посилаються на користувача. Це робить зв'язок "історично правдивим" незалежно від поточного місцезнаходження користувача.

Моделі, що потребують правки:
*   `Transaction` (relation `user`)
*   `ActivityLog`
*   `Comment`
*   `IssueReport`

## 5. План реалізації (Roadmap)

### Етап 1: Підготовка БД та Безпека зв'язків
1.  **Міграція:** Створити таблицю `park_user` (pivot table: `user_id`, `park_id`) для зберігання списку *дозволених* парків.
2.  **Fix Relations:** Оновити методи `belongsTo(User::class)` у моделях (`Transaction`, `ActivityLog` тощо), додавши `->withoutGlobalScope(ParkScope::class)`.

### Етап 2: Логіка User
1.  Додати зв'язок `allowedParks()` у модель `User`.
2.  Додати метод перевірки прав `canSwitchToPark($id)`.

### Етап 3: API
1.  Створити endpoint (наприклад, в `AuthController` або `ProfileController`) для зміни парку.
    *   Перевіряє наявність запису в `park_user`.
    *   Оновлює `users.park_id`.

### Етап 4: Frontend
1.  Отримувати список доступних парків при логіні.
2.  Реалізувати UI перемикач.
3.  При успішній зміні — перезавантаження сторінки/стану (новий контекст підтягнеться автоматично).

## 6. Примітки щодо трансферу Водіїв/Авто
Механізм зміни `park_id` працює і для трансферу ресурсів, але вимагає додаткової бізнес-логіки:
*   Закриття активних змін/закріплень (`VehicleAssignment`) перед трансфером.
*   Обнулення або перенесення балансу (старі транзакції залишаються в старому парку).
